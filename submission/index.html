In this MMSys Grand Challenge, the contestants are asked to design and develop the core function to complete an android app program that downloads a given video file from multiple servers (data nodes) simultaneously. We provide the basic framework of the app and necessary APIs for server querying and connection initiation, so that the contestants can focus on the core transport algorithm, which is wrapped in one function.

Simulator

Each video downloading task starts with passing the basic information (including video ID, file length, video duration, etc.) of the video file to download into the transport function as parameter. The function should first call the API “GET_NODE_LIST” to obtain the list of available data nodes that has the specified video file in storage. The returned list contains no more than 25 data nodes. Note that the returned data nodes are ranked based on the estimated performance of each data node but it may not match the actual performance. You may call “GET_NODE_LIST” API again during downloading, but you are not guaranteed to receive different nodes.

Upon receiving the data node list, you may start connect to each node by calling “CONNECT” API. This API sets up a UDP connection with the specified data node and completes the authentication process. Note that “CONNECT” may fail if the data node is not available or the network is unstable.

Once the data node is connected, you can start sending data request using API “REQUEST”. Each “REQUEST” call can request one or eight consecutive chunks of video data from the specified data node. In this system, all parties agree to divide the video file into small chunks of 1024 bytes (the last chunk of a file may contain less than 1024 bytes) thus we can use chunk ID to represent the small segment of video data in the following context. Chunk size adjustment is not supported in this contest. Upon receiving the “REQUEST”, the data node will put the requested chunk/chunks into the sending queue. The sending queue of the data node has no pacing or congestion control. The sending speed depends on the processing capability of the data node. Note that the “REQUEST” can only be set for one or eight consecutive chunks. If you need to request less than eight consecutive chunks or multiple non-consecutive chunks, you need to call “REQUEST” API multiple times for each chunk you need. You may choose to make multiple “REQUEST” calls continuously without any gap.

The implementation of “REQUEST” uses non-blocking I/O, which means the API does not block the program flow. The the requested data chunk arrives, the program will be notified the chunk ID of the received data chunk and the data node from which the data comes from through a call back function. You are responsible to add necessary logic in the call back function to manage the progress of downloading and maintain data synchronization. You may also need to keep the logic lightweight since each data chunk triggers one call back. One “REQUEST” can ask for eight chunks and may result in eight call backs. Please note that the provided APIs do not monitor the data loss. It is your responsibility to track the possible chunk loss and resend “REQUEST” for the missing chunk. Once you determine the video file has been received in full, you shall exit the function with the success code. You may also exit with the error code if you determine it is unable to finish downloading.

Check documentation for more info. If you get any problems when using simulator, you can find the answer in troubleshooting or you can ask me on GitHub.
